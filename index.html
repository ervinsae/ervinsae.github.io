这两年，关于”AI 会不会取代程序员”的讨论几乎已经被说烂了。每隔一段时间，总会有人抛出一个更刺激的判断：程序员将被大规模替代，写代码会像流水线一样被模型接管，未来只需要少数人发号施令，大多数人都会被清理出场。

这种说法听起来很有冲击力，但我越来越觉得，它真正制造的不是洞见，而是情绪。因为它把一个复杂问题，粗暴地压缩成了一个耸动结论：AI 会写代码，所以程序员要完。

可问题是，程序员如果只是”写代码的人”，那这个职业从一开始就被理解错了。

代码从来都不是软件开发最难的部分。代码只是最后被看见的部分，是冰山露出水面的那一截。真正决定一个系统质量的，往往不是”这一段代码怎么写”，而是”为什么要写这段代码””它解决的到底是不是真问题””它和旧系统如何衔接””它的代价由谁承担””出事之后谁来收拾”。如果把程序员理解成一个高级打字员，那 AI 确实已经足够危险；但如果承认程序员的核心工作从来都包含理解、拆解、权衡、取舍和负责，那问题就完全不是一个层面了。

随想（三）

发表于 2025-12-11 | 分类于 读书
字数统计: 495 | 阅读时长 ≈ 1

在职场中呆久了，接触到不同的人，他们的做事风格各异，这种风格可能和性格，环境以及过往经验有关。

这些做事风格在处理事情上无论是过程还是结果都可能千差万别。我总结大概有两种类型，一种是对上管理型，一种是处理问题型。

虽然对上管理型最终也是要解决问题的，但是他是面向上级解决问题，在解决问题的时候不太会思考怎么解决问题，而在意的是解决问题的结果。这种可能问题最终是被解决了，但是可能不是最佳方式，可能是临时解决，基于这种对上解决问题的方式也注定他在处理问题时不会考虑相对长远的事情，属于领导发现问题解决问题，再发现再解决。这种方式可能比较高效，但是对组织是否有益可能不一定。问题如果都是单点式的处理，就不能形成问题-解决-总结-经验-避免下次再出现的闭环。导致的结果是，可能相同的问题重复出现。

处理问题型其实也就是面向问题解决问题，有可能领导发现的是A问题，但是分析下来其实A是由于BCD问题导致的，其实你可以只解决A，领导的问题被解决了，但是BCD还在那里。因为这种处理问题的方式你会思考问题本身，所以你会总结经验，因为很多问题有相似性。这种方式相对处理问题会低效，但是可能对组织的长远发展是有易的。前期问题比较多，但是一旦处理好之后问题会越来越少，因为每一次的问题处理都是对一类问题的处理。

关于Kotlin中副作用的理解

发表于 2025-11-12 | 分类于 技术
字数统计: 952 | 阅读时长 ≈ 3

review代码的时候看到一个副作用，有点忘记原理了，就问了下AI，我觉得它回答的非常好，记录一下：

提问：不太明白这个副作用函数的运行机制，你上面说它会开启一个协程来执行内部代码，但是又会监听LaunchedEffect中的key值，这个是怎么做到的？另外，如果key值变化了，主线程中正在执行的代码会怎么样？请作为Android资深工程师用浅显易懂的方式给我讲清楚。

AI回答：
现在我们把上面的比喻套在你的代码上：

LaunchedEffect(boValue, countDownTime) { … }

  1. 首次运行：
  • 你的界面第一次显示时，boValue 可能是 0，countDownTime 是 30。
  • LaunchedEffect 记录下 (key1=0, key2=30)，并启动一个协程来执行 {} 里的代码。
  • 这个协程现在开始运行，它会检查 if 条件，发现都不满足，然后就“挂起” (suspend)，等待下一次 key 的变化。
  1. 当 countDownTime 变化时：
    • 1秒后，countDownTime 变成了 29。这导致了界面重组 (Recomposition)。
    • 在重组过程中，LaunchedEffect 会再次被执行。它对比新旧 key：
      • 旧 key: (boValue=0, countDownTime=30)
      • 新 key: (boValue=0, countDownTime=29)◦它发现 countDownTime 变了！于是：
      • 立即取消上一个 countDownTime=30 时启动的协程。
      • 启动一个全新的协程，并记录下新的 key (0, 29)。这个新协程又从头开始执行 {} 里的逻辑。这个过程会每秒重复一次。
  2. 如果 boValue 突然变化了：
  • 假设在倒计时到 25 秒时，血氧传感器返回了数据，boValue 从 0 变成了 98。
  • 界面再次重组。LaunchedEffect 对比 key：
    • 旧 key: (boValue=0, countDownTime=25)
    • 新 key: (boValue=98, countDownTime=25)
  • 它发现 boValue 变了！于是：
    • 立即取消上一个 (0, 25) 的协程。
    • 启动一个全新的协程。这个新协程一进来就检查 if (boValue > 0)，条件满足，立刻执行 navigator.navigate(…) 进行页面跳转。任务完成！

关于主线程的问题

你还问到：“如果key值变化了，主线程中正在执行的代码会怎么样？”这是一个非常好的问题，涉及到协程的核心优势。
  • LaunchedEffect 里的代码默认运行在主线程上，但它是可挂起 (suspendable) 的。
  • 当协程执行到一个挂起点（比如 delay(1000) 或者其他需要等待的异步操作）时，它会释放主线程，但并不会销毁自己。主线程可以去干别的事，比如响应用户点击、刷新动画等，UI 不会被卡住。
  • 当 key 变化时，是 Compose 框架在主线程的重组阶段发现了 key 的变化，然后它会向旧的协程发送一个取消信号。
  • 正在运行的协程（无论它当前是真正在执行 CPU 指令，还是正处于挂起状态）收到取消信号后，会在下一个挂起点或检查点响应取消，并抛出 CancellationException 来终止自己。
  • 这一切的取消和重启调度都是由 Compose 框架和 Kotlin 协程库在背后无缝完成的，不会阻塞主线程的正常工作。

总结

  • 机制：LaunchedEffect 不是“边运行边监听”，而是在每次重组时，通过对比新旧 key 值来决定是否取消旧协程，启动新协程。
  • 目的：确保副作用代码（比如网络请求、数据库操作、导航）始终与当前的 state (也就是 key) 保持同步。
  • 线程：代码虽然在主线程协程中，但由于其“可挂起”的特性，不会阻塞 UI 渲染。它的生命周期管理是自动且安全的。
